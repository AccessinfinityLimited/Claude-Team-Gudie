Daily Git & Firebase Workflow

Pull → push → deploy — do this every day

›
💡 Good habit: Always pull first → make changes → push → deploy. Never push without pulling first to avoid conflicts.

Always start here — get the latest version from remote.

⬆ Push Your Changes

Stage, commit, and push your work.

git add .

git push origin main

🚀 Deploy to Firebase

Ship the latest code live after pushing.

firebase deploy

/streamlit-app

Turn any Python script into a clean Streamlit web app.

✅ No install needed. Enabled for every @accessinfinity.com account in Claude Desktop. Just type /streamlit-app and your request.
i

What it does

Python script → usable web app, no Streamlit knowledge needed

›
  • Wraps a Python script in a usable web UI — no Streamlit knowledge needed.
  • Always follows the same shape: Input → Process → Output → Download.
  • Keeps your original logic as plain functions; adds validation, spinners, and friendly errors by default.
  • Best for: file-upload → process → download tools, dashboards, sharing scripts with non-coders.
→

How to use it

Four steps from script to running app

›
  • Open Claude Desktop and a folder containing your Python script.
  • Type /streamlit-app followed by what you want.
  • Review the plan Claude proposes (inputs, outputs, downloads) and confirm.
  • Run the generated app with the two commands below.
Example prompts 
/streamlit-app Turn sales_report.py into an app where I can upload a CSV and download the report.

/streamlit-app Build an app that takes a PDF, extracts tables, and lets me download them as Excel.

/streamlit-app Make a small UI for the forecast() function in model.py.
Run the generated app 
pip install -r requirements.txt
streamlit run app.py
💡 Tip: Tell Claude who will use it ("for the ops team, not engineers") and list inputs/outputs upfront — saves a round of clarifying questions.

/firebase-poc-scaffold

Spin up a new POC on the standard React + Firebase stack.

✅ No install needed. Enabled for every @accessinfinity.com account in Claude Desktop. Just type /firebase-poc-scaffold and your request.
i

What it does

One known-good baseline for every new prototype

›
  • Scaffolds a fresh POC app on the company's standard stack: React (Vite + TS), Tailwind, shadcn/ui, Firebase (Auth, Firestore, Functions, Storage).
  • Wires the Firebase Emulator Suite so you can build locally with no cloud project.
  • Includes routing with React Router, lazy-loaded pages, and a ProtectedRoute wrapper.
  • Auth model: email/password against the emulator locally; switches to Microsoft (Azure AD) auth once you connect a real Firebase project.
  • Best for: starting any new prototype, demo, or MVP — so every project follows the same layout.
→

How to use it

From zero to a running local app

›
  • Make sure the Quick Start tab tools are installed (Node 18+, Git, Firebase CLI).
  • Open Claude Desktop in the parent folder where you want the new project to live.
  • Type /firebase-poc-scaffold and tell Claude the app name (e.g. "task-tracker").
  • Claude checks prerequisites, scaffolds the project, and runs lint + build to verify it's clean.
  • Start local dev with npm run dev:full — it boots Vite + the emulators together.
Example prompts 
/firebase-poc-scaffold Create a new POC called "task-tracker".

/firebase-poc-scaffold Spin up a fresh demo app for the sales team — emulator-only for now.

/firebase-poc-scaffold Bootstrap a new app and wire it to my existing Firebase project.
Run locally after scaffold 
npm run dev:full   # Vite + Firebase emulators
# Open the app:           http://localhost:5173
# Open the Emulator UI:   http://localhost:4000
# Test login:             test@example.com / password123
⚠️ Secrets: VITE_FIREBASE_* values go in .env.local (git-ignored). Sensitive third-party keys (Stripe, SendGrid, etc.) belong in Firebase Secret Manager, never in a VITE_ var.

/firebase-app-review

Run a full review of a Firebase app — code + browser.

✅ No install needed. Enabled for every @accessinfinity.com account in Claude Desktop. Just type /firebase-app-review and your request.
i

What it does

One checklist, same bar for every Firebase app

›
  • Reviews any Firebase app built on the standard stack against the same checklist every time, so nothing important slips through.
  • Two modes: static code review (a file/component or the whole app) and live browser test (drives a running app, watches console + network).
  • Leads with security: open Firestore rules, missing ownership checks, leaked secrets, broken auth — flagged as Critical / High.
  • Reports findings in a fixed structure (Summary → Findings by severity → What looks good → Next steps) — easy to scan and compare across apps.
  • Best for: before a demo, after a big change, or any time you want a second pair of eyes on a POC.
→

How to use it

Point Claude at what to review and read the report

›
  • Open Claude Desktop in your project folder.
  • Type /firebase-app-review and tell Claude what to look at: a file, a component, the whole app, or a URL.
  • For a browser test, start your app first (npm run dev:full) so Claude can drive it at http://localhost:5173.
  • Read the report top to bottom — fix anything Critical or High before sharing the app.
Example prompts 
# Review a specific file
/firebase-app-review Look at src/lib/firebase.ts and firestore.rules.

# Full review (static + browser)
/firebase-app-review Do a full review of this app before tomorrow's demo.

# Live browser test only
/firebase-app-review Test the app running at http://localhost:5173 — try sign-up, the main flow, and sign-out.
⚠️ The #1 thing this catches: Firestore rules left as allow read, write: if true. A POC with open rules is the most common and most damaging issue — always run this before sharing a link.
